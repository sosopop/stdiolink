# Update Knowledge Base

## Purpose

约束何时更新知识库，以及如何用最小动作补齐正常更新和反思型更新。

## Use When

- 功能、协议、API、元数据、调试路径或测试入口发生变化。
- 一次真实问答、排障、开发过程暴露出更短的知识路径。

## Routine Update

1. 先更新受影响的主题文档，优先补“修改入口、约束、测试入口”。
2. 再更新对应目录 `README.md`，确保索引能找到新内容。
3. 只有通用问题路由确实缺失时，才更新根索引 `doc/knowledge/README.md`。

## Reflection Update

1. 先提炼本次过程的最短路径：先读什么、先改哪里、先跑什么测试。
2. 只把高频误区、稳定排障顺序、缺失入口写回知识库，不重复记录问题答案本身。
3. 反思内容只对单模块有效时回写子系统文档；跨子系统路径再放 `08-workflows/`。

## Placement Rule

- 子系统内规则放对应一级目录；跨子系统实现路径放 `08-workflows/`。
- 测试/构建/调试通用路径优先放 `07-testing-build/`。
- 长背景说明留在 `doc/manual/`，不要把知识库写回成长手册。

## Writing Rule

- 每条经验尽量控制在 1 到 3 行，只点明注意项、简单约束和入口。
- 知识库正文只写结构、流程、入口和约束，不贴具体代码；代码文件是最终事实源。

## Minimum Checklist

- 是否影响实现路径判断、联动关系或测试/调试入口。
- 对应目录 `README.md` 和根索引 Query Routing 是否仍能准确检索到。
